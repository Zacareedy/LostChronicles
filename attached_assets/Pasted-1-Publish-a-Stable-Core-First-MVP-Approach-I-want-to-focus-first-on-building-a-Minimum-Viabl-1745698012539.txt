1. Publish a "Stable Core" First (MVP Approach)

I want to focus first on building a Minimum Viable Product (MVP):

MVP Features:

Basic terminal system (login, file exploration, basic commands: ls, cd, cat, help, clear, etc.).

Core map component (zooming slightly, clicking on stations [even if interaction is limited at first]).


Very limited lore exposure (only hints — not full documents or mysteries yet).

Error handling: broken commands should return realistic "error" messages rather than crashing.

Important:

The app must be fully stable and playable even if later puzzles and systems are not yet active. Non-implemented commands, map nodes, and features should show "ACCESS DENIED" or "SYSTEM MODULE OFFLINE" messages until added later.

2. Structure for Easy Updates

I need the app to be modular:

Modular design rules:

New files, maps, and terminal commands should be easily addable without breaking old ones.

Locked features (like the SubNet Interface, Radio, Protocol Candle, etc.) should exist in the code but are hidden or disabled until activated by devs like me through future updates.

Map markers should allow "future expansion" without needing full rebuilds (like pulsing unknown areas you can activate later, “redacting” most of the map until its features are fully implemented in a future app update).

Puzzle triggers (like entering the right command) should be handled by a central logic file so you can easily add more without redoing everything.

Important:

You need to build a dev-only "hidden features toggle system" — where admins (me) can flip a feature on/off when ready, instead of coding it fresh every time (eg the radio transmission puzzle and black box puzzle are still in development, so should remain in the code but NOT be accessible to users until I choose to update the app. Similarly, I do not want any audio for the app at the moment, but I still need to keep that functionality to enable it when it is fully polished)

3. Lore Expansion Handling

I plan to drip-feed new lore over time. To support that:

Lore system must allow:

New text files being easily added to the file system.

New hidden folders appearing after certain milestones.

New radio signals, map locations, or hidden stations appearing dynamically (not static hardcoded only).

Subtle environmental changes (example: glitched screens, background flickers, changes to map terrain) after updates without needing user downloads.

4. Bug Policy for Early Version

I accept small non-critical bugs at launch (example: minor map zoom issues, flicker bugs) as long as:

No major crashes.

Core commands (ls, cd, cat, help, etc.) work reliably, as well as the system failure sequence (epecially reliably stopping it when entering the numbers).

No data loss (i.e., user progress is remembered — if possible via cookies/localStorage if user accounts aren't implemented yet).

Any unfinished features are gracefully gated behind "ACCESS RESTRICTED" messages.

This approach matches how real ARGs often feel — slightly broken but intentional.
